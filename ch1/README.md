## 什么是并行

我们的 CPU 有多个核心。举个例子，假设我们要计算下面这个式子：

```text
((11 + 21) * (31 + 41)) + (51 + 61)
    式1          式2          式3
```

现在我们有三位小学生。我们可以把三个小算式分配给三位小学生，每个人只需要先算一次加法，比如 `11 + 21`。

同时我们也能观察到，`式1`、`式2`、`式3` 之间其实存在执行顺序：
- `式1` 和 `式2` 可以先分别算出来
- 然后再把 `式1` 和 `式2` 的结果相乘
- 最后再把这个乘积与 `式3` 的结果相加

也就是：

- `a`：同时计算 `式1`、`式2`、`式3`
- `b`：计算 `式1` 和 `式2` 的乘积
- `c`：将步骤 `b` 的结果与 `式3` 的结果相加

当问题规模上升，比如变成 1000 个算式时，我们也可以找 1000 个小朋友分别计算，最后再按规律把结果组合起来，得到最终答案。

在之后的实验中，我们会遇到“必须按照某种顺序处理结果”的内容。当然，这不是 `ch1` 需要考虑的。

把 1000 个简单算式分配给 1000 个小学生分别处理，这个过程就可以理解成 **并行**。

而在操作系统中，“小学生”对应的概念就是 **线程**。我们可以创建线程，把任务交给它去做；等它完成之后，再做后续处理。
例如，在 `pthread` 库中，`pthread_join()` 常用来等待某个线程执行结束。

到这里，你应该已经对“并行”有了一个比较直观的概念。

在编程时，我们经常会遇到这样的问题：

```c
for (int i = 0; i < N; i++) {
    // 对 data[i] 进行某种操作
}
```

如果 `data[i]` 之间彼此独立，那么我们是不是可以召唤几个“小朋友”（也就是创建几个线程），把这些 `data[i]` 分成几份，交给不同的人去处理？

但每次都手写 `pthread_create()`，再手动把任务分配给不同线程，往往会比较麻烦，也比较单调。

于是，**OpenMP** 扩展就诞生了。

## OpenMP 是什么

比如：

```c
#pragma omp parallel for reduction(+:sum)
for (int i = 0; i < N; i++) {
    sum += i * 0.1;
}
```

我们先不看 `parallel for reduction(+:sum)` 这一整串。

OpenMP 不是像 Python 中 `import` 那样直接导入一个库来使用。这里的 `#pragma` 更准确地说，是一种“给编译器看的指令”。

当编译器看到：

```c
#pragma omp parallel for
```

它就知道：这里的 `for` 循环可以尝试并行执行，也就是自动帮我们安排多个线程，把循环任务分配出去。

`#pragma omp parallel for ...` 后面还可以跟很多内容。那这里的 `reduction` 又是什么意思呢？

## reduction

`reduction` 在编程里一般可以理解成 **“合并”或者“归约”**：把很多个局部结果，按某种规则合成一个最终结果。

一说到“合并”，很多人脑海里会立刻想到加、减、乘、除。比如：
- 把所有值依次加起来
- 把所有值依次乘起来
- 取最大值
- 取最小值

没错，OpenMP 里也是类似的思路。

我们看括号里的 `+:sum`。

先看 `:` 后面的 `sum`。这里的 `sum` 只是变量名，也就是最终结果要写回哪个变量。

同样地，我们也可以写成：

```c
#pragma omp parallel for reduction(+:aa)
for (int i = 0; i < N; i++) {
    aa += i * 0.1;
}
```

你看，改成 `aa` 也没有问题。

那 `:` 前面的 `+` 又是什么意思？

我们刚才说过，`reduction` 这种“合并操作”，必须先知道自己到底是按什么规则在合并。比如是做加法、乘法，还是取 `max` 或 `min`。

而这种规则，其实要和循环体内部的操作一致。也就是说，像：

```c
aa += i * 0.1;
```

这里是“加法累计”，那么 `reduction` 里就应该写成 `+`。

一般来说，写给 OpenMP 的 `#pragma` 指令中的归约操作，也就是 `:` 前面的这个符号，需要和循环体中的操作保持一致；否则结果可能出错。

为什么会出错呢？

我们可以继续用“小学生”的类比来理解。

编译器在分配任务的时候，通常会给每个线程准备一个“局部版本”的变量。你可以把它想成：每个小学生都有一张自己的草稿纸，先把自己算出来的中间结果记在草稿纸上。这里我们先把这张草稿纸记作 `aa'`。

那这张草稿纸一开始应该写什么？

如果每个线程内部做的是：

```c
aa += i * 0.1;
```

为了保证最后合并出来的结果正确，那么每个线程自己的局部变量就应该从 `0` 开始。

因为对于加法来说，`0` 是最自然的起点，`0 + x = x`。

同理，如果做的是乘法，那么局部变量通常就应该从 `1` 开始，`1 * x = x`。

决定这个局部变量初值应该是什么的，正是 `reduction` 中 `:` 前面的那个符号。

## schedule

除了 `reduction`，OpenMP 还有什么别的选项吗？
有的，兄弟，有的！

```c
#pragma omp parallel for schedule(...)
```

`reduction` 是对一个数列的每一项做一个合并操作，合并运算就是加/乘/min/max 这种。除此之外，如果我们需要对一个物体的多个分块同时处理，就用不了 `reduction` 了。你想想，一张图片虽然被分块处理了，但是最后的结果还是一张图片，这其中并没有 `reduction` / 合并 / fold 这样的操作。

`schedule` 用来决定循环迭代如何分配给不同线程。在当前实验里，如果每次循环刚好对应一个任务块，那它也就相当于决定“这些块如何分给线程”。

我们可以怎么用 `schedule` 呢？

首先，`schedule` 可以接收几种参数：`static` / `dynamic` / `guided` / `auto`。下面我们具体讲讲这几个参数是干什么的，怎么使用。

首先从 `static` 开始，具体用法是：

```c
#pragma omp parallel for schedule(static)
for (...) {
    // 你想做的操作
}
```

如果有 4 个线程、20 个任务，那么每个线程就会拿到 5 个任务，很平均。

或者写成：

```c
#pragma omp parallel for schedule(static, 8)
```

这里的 `8` 表示每个 `chunk`（块）里有 8 个任务，也可以理解成线程每次被分配到的任务块大小是 8。

同样是 4 个线程、20 个任务，那么会先形成大小分别为 `8`、`8`、`4` 的几个任务块，再把这些块分配给线程。你可以先粗略理解成：前面是两个 8，最后剩下一个 4。

`static` 会遇到什么样的问题呢？虽然我们把任务量均匀分配了，但是每个任务的长短不一。可能线程 1 分配到的任务难度大，需要运行的时间长，而线程 2 的任务很简单。类似就是一个小学生算 `1239 + 29384`，另一个算 `11 + 12`，差别可想而知。最后，整个程序会被最慢的那个线程拖住。

解决这样的问题，我们就需要用 `dynamic`。

```c
#pragma omp parallel for schedule(dynamic)
```

你可以粗略理解成：每个线程先取一部分任务，谁先做完手上的任务，谁就更早拿到下一部分任务。这样时间分配就相对更合理。

这里要引入一个新概念：**临界资源**。所谓“临界资源”，意思是当前这个资源同时只能有一个人访问或使用。思考这样一个情况：室友和自己一共四个人，同时回到寝室，大家都想先洗澡然后上床睡觉，但是卫生间这个资源只有一份，同时只能容纳一个人洗澡，剩下的人要排队等待。这时，“卫生间”就是临界资源。

回到 OpenMP，你可以把后续任务的分配过程粗略理解成一种共享资源访问：多个线程都想去拿下一部分任务，但同一时刻总会有协调和分配的开销。所以可能出现这样的情况：4 个甚至 8 个线程同时完成工作，但是不可能所有人都在同一时刻零成本地拿到下一部分任务，这样就会有一些额外的时间消耗。

为了解决这个问题，我们可以让线程事先拿 8 份任务。这样多个线程处理完 8 个任务的时间通常不会完全相同，大家错峰访问任务队列，就能减少额外的时间消耗。

回到代码，我们在 `for` 循环之上加：

```c
#pragma omp parallel for schedule(dynamic, 8)
```

就能做到。`ch1` 的实验中有具体的例子。

那么什么是 `guided` 呢？

大概就是：任务数量一开始有 100 个，这个时候 `(dynamic, 8)` 还不算太大；但是任务只剩 20 个的时候，`(dynamic, 8)` 就和 `(static, 8)` 没有太大差别了。而 `guided` 的意思就是系统能自动根据剩余任务的数量，减少线程每次的工作量。类似于：任务很多的时候像 `(dynamic, 8)`，任务快做完的时候就自动变成更小的块。

`auto` 最简单，什么都不用管，交给编译器自动决策。
